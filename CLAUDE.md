## Development workflow

When you start work on a GitHub issue, **set its status to "In Progress" on the project board before writing code**, and create a `feature/issue-<n>-<slug>` branch. Status is tracked on the **GitHub Projects (v2) board "Toastily for CN Collaborators"** (owner `geldan01`), not via labels — the issue must be an item on the board with its **Status** field set.

- Operate only as the `geldan01` account (`gh auth switch --user geldan01` if needed) — it owns the repo and the board. Don't use other authenticated accounts.
- Setting the Status field requires the `project` token scope. If `gh` lacks it, the user must run `gh auth refresh -s project --hostname github.com` themselves **while the browser is signed in as geldan01** (interactive flow — you cannot drive it, and authorizing under the wrong account grants the scope to the wrong token).
- Stable IDs for the board (re-list if they ever change):
  - Project: number `5`, id `PVT_kwHOAprzFs4BaND7`
  - Status field id: `PVTSSF_lAHOAprzFs4BaND7zhVGURU` — options: Backlog `6ca1ef53`, Todo `f75ad846`, **In Progress `47fc9ee4`**, Done `98236657`
- Find the issue's item id, then set Status:
  ```sh
  gh project item-list 5 --owner geldan01 --format json --limit 200   # find the item id for the issue (content.number)
  gh project item-edit --id <itemId> --project-id PVT_kwHOAprzFs4BaND7 \
    --field-id PVTSSF_lAHOAprzFs4BaND7zhVGURU --single-select-option-id 47fc9ee4
  ```
  (Add the issue to the board first with `gh project item-add 5 --owner geldan01 --url <issueUrl>` if it isn't already an item.)
- Move the item to **Done** (`98236657`) when the work merges.

### Scheduling, numbering & executive RBAC (extends P3)
- **Executive positions/assignments** ([server/db/schema/executives.ts](server/db/schema/executives.ts), PRD §3.2, issue #47): admin-managed positions carry **one write-access boolean per executive-hub group** (`write_people`, `write_meetings`, `write_content`, `write_communication`, `write_config`) — authority is data, never hard-coded against a position name. Assignments are temporal (current holder = `ended_at` null; history kept) and **never change a position's write access** — reassigning a role to a new member leaves the matrix untouched. Assigning a position **promotes a member → officer** with a role-history entry. Seed: standard 7 positions; President = all five groups, VP Education = meetings, VP PR = content + communication, Secretary = meetings. Position names/active state managed at [/admin/executives](app/pages/admin/executives.vue) (admin); **write access is edited as a positions × groups matrix on the Permissions page** ([/admin/permission-grants](app/pages/admin/permission-grants.vue), admin-only edit; officers/President see it read-only). The site admin always manages every group and is **not a row** in the matrix.
- **Effective capabilities** are the single source of truth: `effectiveCapabilities(user)` ([server/utils/executives.ts](server/utils/executives.ts)) = admin (all) ∪ caps derived from current exec-position **group write-access** ∪ delegable per-user grants. The six derived caps (`canManageCalendar`/`canManageMinutes` ← meetings, `canManageContent` ← content, `canAssignOfficers` ← people, `canManageCommunication` ← communication, `canManageConfig` ← config) each gate their group's endpoints — `requireCalendarManager`/`requireContentManager`/`requireCommunicationManager` ([server/utils/communication.ts](server/utils/communication.ts))/`requireConfigManager` ([server/utils/club-config.ts](server/utils/club-config.ts)); **notifications are communication-gated and settings config-gated** (no longer plain officer/admin). All groups stay **readable** by every officer — the matrix governs **write** only. Exposed to the client at `GET /api/me/capabilities` via [useCapabilities()](app/composables/useCapabilities.ts); the `calendar`/`content`/`communication`/`config` route middleware + the executive-hub cards gate on these.
- **Meeting lifecycle:** `meetings.status` (`scheduled`/`cancelled`) + `meeting_number` + `minutes_en/fr` (minutes UI later). **Cancel** keeps the row (struck-through, no number); **holidays delete** any meeting on that date. **Numbering is auto & contiguous** — `renumberMeetings()` ([server/utils/meetings.ts](server/utils/meetings.ts)) numbers non-cancelled meetings in date order from the `meeting.number_start` setting, called after every add/delete/cancel/generate/holiday change.
- **Generate-year:** `POST /api/admin/meetings/generate` ({firstDate, untilDate, everyWeeks, numberStart, location}) creates recurring meetings, skipping holidays and existing dates. UI on [/admin/meetings](app/pages/admin/meetings.vue) defaults to the upcoming Jul 1 → Jun 30 span.
- **Per-meeting signup authority (PRD §3, contextual):** a meeting role can carry `grants_meeting_authority` ([meeting_roles](server/db/schema/meetings.ts)); the member signed up for such a role on a given meeting becomes a **meeting manager** for that meeting — independent of executive rank. `isMeetingManager(user, meetingId)` ([server/utils/meeting-authority.ts](server/utils/meeting-authority.ts)) = officer/admin **OR** holds an authority-granting role on that meeting. Managers can **assign any member or a guest, reassign a filled role/speech slot, release anyone's signup, and edit any speech title**; plain members may only self-claim open roles and release their own. Enforced in `signup.post/delete` + `speech.post/patch/delete`; surfaced to the client as `canManageSignups` on `GET /api/meetings/[date]`, which drives the **Assign/Reassign dropdown** ([MeetingAssignPanel.vue](app/components/MeetingAssignPanel.vue), member-or-guest) on [/meeting/[date]/signup](app/pages/meeting/[date]/signup.vue). The dropdown's roster comes from `GET /api/meetings/members` (any logged-in member, id+name only). Authority is **data, never a hard-coded role name** — admins toggle it per role at [/admin/meeting-roles](app/pages/admin/meeting-roles.vue); seed grants it to the Toastmaster. This same flag will back the SAA/Toastmaster voting controls in §8.

### Image uploads (§2/§15, issue #10)
- **S3-compatible object storage**, backend-agnostic — the same code drives **MinIO** (self-hosted on Coolify), **Cloudflare R2**, **Backblaze B2**, or AWS S3; only env differs. Config lives in `runtimeConfig.s3` ([nuxt.config.ts](nuxt.config.ts)), sourced entirely from `S3_*` env (Coolify-injected, never committed): `S3_ENDPOINT`/`S3_REGION`/`S3_BUCKET`/`S3_ACCESS_KEY`/`S3_SECRET_KEY`, `S3_FORCE_PATH_STYLE` (true for MinIO/self-hosted), `S3_PUBLIC_BASE_URL` (optional), `S3_MAX_BYTES` (default 5 MB). Empty `S3_ENDPOINT` ⇒ uploads disabled and the route returns **503** (graceful per-deployment opt-in).
- **Helpers** in [server/utils/s3.ts](server/utils/s3.ts): `isStorageConfigured`, `validateImageUpload` (type allow-list + size → canonical ext), `generateImageKey` (`uploads/<yyyy>/<mm>/<uuid>.<ext>`, UTC-partitioned, unguessable), `publicUrlForKey` (direct bucket URL when `S3_PUBLIC_BASE_URL` is set, else the Nitro proxy), and the memoised `useS3Client`.
- **Routes:** `POST /api/uploads` ([server/api/uploads/index.post.ts](server/api/uploads/index.post.ts)) — gated by the **content-edit capability** via `requireContentManager` ([server/utils/content.ts](server/utils/content.ts), mirrors `requireCalendarManager`); accepts a multipart `file`, validates, stores, returns `{ key, url }`. `GET /api/uploads/[...key]` ([server/api/uploads/[...key].get.ts](server/api/uploads/[...key].get.ts)) — **public** proxy that streams an object from the bucket for private-bucket deployments (guards the `uploads/` prefix; only used when `S3_PUBLIC_BASE_URL` is unset).
- **Local dev:** `docker compose up -d minio minio-init` runs MinIO + a one-shot that creates the bucket and makes it publicly readable; the bundled `.env.example` S3 defaults point at it. **Tests:** unit pin the pure helpers ([tests/unit/s3.test.ts](tests/unit/s3.test.ts)); integration pins the auth boundary + 503 degradation ([tests/integration/uploads-api.spec.ts](tests/integration/uploads-api.spec.ts), with storage forced off via `S3_ENDPOINT=''` in [playwright.config.ts](playwright.config.ts)).

### Participation & attendance (§11, issues #11/#35)
- **Read-side aggregation** in [server/utils/participation.ts](server/utils/participation.ts) (no destructive updates — history is built from existing signup/speech/vote/role records): a club-wide `participationSummary` (one row per member: attended/roles/speeches/evaluations/awards counts) and a per-person `memberParticipation` timeline. Surfaced member-gated at `GET /api/participation` + `/api/participation/[id]`, rendered at [/participation](app/pages/participation/index.vue) (+ `[id]`).
- **Member attendance** (issue #35) is the member equivalent of guest check-in: `meeting_attendance` ([server/db/schema/meetings.ts](server/db/schema/meetings.ts)) — **one row per member per meeting** (unique `meeting_id`+`user_id`), with `source` (`self`|`secretary`) and `recordedBy`. Two entry paths mirror role-signup authority: a **member self-checks-in** (`source: 'self'`, `recordedBy: null`), or a **meeting manager / secretary records others** (`source: 'secretary'`) — gated by `isMeetingManager`, **never a hard-coded "Secretary"** (we track *who* recorded it instead). Endpoints: `POST`/`DELETE /api/meetings/attendance` and member-gated `GET /api/meetings/[date]/attendance` (present list + `canManage` + `selfPresent` + a **present count** = members present + guest check-ins, the quorum aid via [server/utils/attendance.ts](server/utils/attendance.ts) `meetingPresentCount`).
- **UI:** [/meeting/[date]/attendance](app/pages/meeting/[date]/attendance.vue) (member-gated) — present-count cards, a self check-in toggle, and for managers a roster with pre-filled present toggles; a one-tap member self check-in on [/checkin](app/pages/checkin.vue) (members see it instead of the guest name form); an **Attendance** button on the meeting page. Attendance feeds both the `/participation` aggregate (`attended` count) and the per-person timeline (**Meetings attended** section).

### Written evaluations (§11, issue #60)
- **Digital peer feedback on speeches**: anyone *checked in* to a meeting — a member (`meeting_attendance`) OR a checked-in guest (`guest_checkins`) — may leave a written evaluation of a prepared speech (broader than the agenda's formal evaluator slot, which is unchanged). Schema `written_evaluations` ([server/db/schema/evaluations.ts](server/db/schema/evaluations.ts)): one row per (`speech_id`, evaluator), evaluator a member (`evaluator_user_id`) **or** guest (`evaluator_guest_name`), with two free-text fields (`liked`/`recommend`) + three required 1–5 star ratings (`structure`/`vocal_variety`/`gestures`). **Editable** — re-submitting updates the existing row (member uniqueness via the `(speech_id, evaluator_user_id)` constraint; guest re-submits deduped by case-insensitive name in the API).
- **Eligibility gate** in [server/utils/evaluations.ts](server/utils/evaluations.ts) (`isCheckedIn`, + the pure `isValidRating`): the submitter must be checked in. Endpoints: **public** `POST`/`DELETE /api/meetings/evaluation` (a logged-in member submits as themselves and can't evaluate their own speech → 409; a guest submits via the name they checked in with), and `GET /api/meetings/[date]/evaluations` (public — the evaluable speeches, plus a member's `selfEligible` + their own `mine` to pre-fill).
- **UI:** [/meeting/[date]/evaluations](app/pages/meeting/[date]/evaluations.vue) (NOT member-gated — guests evaluate too; guests enter their check-in name) with a reusable [StarRating.vue](app/components/StarRating.vue); an **Evaluations** button (everyone) on the meeting page. **The speaker reads back the feedback they received** in their participation timeline (**Evaluations received** on [/participation/[id]](app/pages/participation/[id].vue)) — **private**: `memberParticipation(..., { includeReceivedEvaluations })` populates it only when the requester is that member or an admin (gated in the endpoint).

### Member profiles (§7/§11, issue #61)
- **Self-authored member directory fields** on `users` ([server/db/schema/users.ts](server/db/schema/users.ts)): `bio`, `goals`, `phone` (all nullable, member-authored free text — **single language, the member's choice**, NOT bilingual DB labels), plus the contact-visibility preference `show_contact_info` (boolean, default true). The **photo** reuses the existing self-service avatar (`avatar_key`, issue #43) — no new upload path. Notification opt-outs already live on `users` (issue #59) and are unchanged.
- **API (member-gated, own row only):** `GET`/`PUT /api/me/profile` ([server/api/me/profile.get.ts](server/api/me/profile.get.ts) / [.put.ts](server/api/me/profile.put.ts)) — PUT changes only the fields present in the body; bio/goals/phone are trimmed (empty ⇒ null); the handler only ever writes the caller's row.

Applied in **two** read paths — the roster `GET /api/members/roster` ([roster.get.ts](server/api/members/roster.get.ts), gated on the viewer) and the per-person timeline via `memberParticipation(..., { includeContact })` ([server/utils/participation.ts](server/utils/participation.ts), the endpoint passes `includeContact` only when the requester is the member or an admin — same shape as `includeReceivedEvaluations`).
- **UI:** an editable **My profile** card on [/account](app/pages/account.vue) (bio/goals/phone + the visibility toggle); the roster ([/members](app/pages/members/index.vue)) renders bio, goals, phone (tel link) and "Contact details hidden" when email is withheld; the participation page ([/participation/[id]](app/pages/participation/[id].vue)) shows an About/Goals/Contact block. **Tests:** integration pins the auth boundary, the bio/goals/phone round-trip (trim/null), and the contact-visibility gate on both the roster and the timeline ([tests/integration/member-profile-api.spec.ts](tests/integration/member-profile-api.spec.ts)).

### Calendar export (ICS) + role reminders (§10/§6, issue #59)
- **ICS export** — pure RFC 5545 helpers ([server/utils/ics.ts](server/utils/ics.ts): `escapeIcsText`/`icsLocalRange`/`foldIcsLine`/`buildIcsCalendar`) + a meeting→event mapper ([server/utils/meeting-calendar.ts](server/utils/meeting-calendar.ts)). Times are **floating local** (no `Z`/`TZID` — there's no club-timezone setting; a local club event is interpreted in the viewer's zone); duration from the `meeting.duration_minutes` setting (default 120). Two endpoints: **public** single-meeting `GET /api/meetings/[date]/calendar` ([calendar.get.ts](server/api/meetings/[date]/calendar.get.ts), 404 on cancelled/missing) and **member-gated** personalised feed `GET /api/meetings/calendar` ([calendar.get.ts](server/api/meetings/calendar.get.ts), upcoming scheduled meetings where the member holds a role signup or speech, deduped). Both honour `?lang=en|fr` and set `text/calendar` + `Content-Disposition: attachment`. **Add-to-calendar** buttons (download links): single on the meeting page, "Add my meetings" on [/meetings](app/pages/meetings/index.vue) (members).
- **Role reminders** — a daily task (`'0 9 * * *'` in [nuxt.config.ts](nuxt.config.ts)) [notifications:role-reminders](server/tasks/notifications/role-reminders.ts) emails members before a meeting where they hold a role/speech. For each scheduled meeting entering the window (today ≤ date ≤ today + `meeting.reminder_days_before`, default 1) with `meetings.role_reminder_sent_at` null, `notifyRoleReminders()` ([notifications.ts](server/utils/notifications.ts)) sends a **personalised** per-recipient email (template `meeting_role_reminder`, placeholders `{{member_name}}`/`{{meeting_date}}`/`{{meeting_time}}`/`{{location}}`/`{{roles}}`/`{{meeting_link}}`) through the existing email pipeline (logged as `scheduled`), then stamps `role_reminder_sent_at` so a meeting is reminded once. **Members opt out** via `users.notify_role_reminders` (default true) — toggled on [/account](app/pages/account.vue), read/written at `GET`/`PUT /api/me/preferences`; guests excluded.
- **Signup reminder (evolution of the `unfilled_roles` weekly reminder, issue #59):** the scheduled "please sign up" email now lists **open roles AND open prepared-speech slots** for the upcoming meeting(s) and links to the meeting **signup** page (`/meeting/[date]/signup`). It honours a per-member opt-out (`users.notify_signup_reminders`, default true; second toggle on [/account](app/pages/account.vue)) — `sendNotification()` special-cases the `SIGNUP_REMINDER_TEMPLATE_KEY` (`'unfilled_roles'`) to use `getSignupReminderRecipients()`; open speech slots come from `getUpcomingUnfilled()` (slots offered = max(`speeches.max_per_meeting`, existing) − speeches with a speaker). It is **configurable by agenda/calendar managers OR communication managers**: the email-**schedule** endpoints gate per-template via `requireScheduleManager(event, templateKey)` (calendar managers may manage only the signup-reminder schedule; every other template stays communication-only via `requireCommunicationManager`), and [/admin/notifications](app/pages/admin/notifications.vue) (now behind the `notifications` middleware = comms **or** calendar) shows calendar-only managers just a focused "Signup reminder" enable/day/time card while communication managers keep the full templates/schedules/log UI. Template editing, send-now, and the log stay communication-only.

**Testing** is layered — **Vitest** for pure logic (`tests/unit`), **Playwright** for both API/integration (`tests/integration`, via the `request` context) and browser e2e (`tests/e2e`). Playwright runs the app on **port 3100** against a **dedicated `*_test` database** (`pnpm test:db:prepare` drops/recreates/migrates/seeds it), with per-role fixtures (`guestPage`/`memberPage`/`officerPage`/`adminPage`/`managerPage`, `apiAs(role)`) backed by seeded `@toastily.test` accounts; email is auto-stubbed (verification tokens read from the DB). **Read [docs/TESTING.md](docs/TESTING.md)** for the full harness, the feature×layer coverage matrix, and how to add tests. CI runs all three layers ([.github/workflows/test.yml](.github/workflows/test.yml)).

**Migrations** run via [server/db/migrate.mjs](server/db/migrate.ts) (`pnpm db:migrate`, also the Docker boot command) — a standalone runner that applies **each migration in its own committed transaction**. This matters: drizzle-kit/drizzle-orm batch all pending migrations into one transaction, which breaks a fresh apply when an enum value is added in one migration (`0003`) and used in a later one (`0012`) — Postgres forbids using a new enum value in the transaction that added it (`unsafe use of new value`). The runner's bookkeeping is byte-compatible with drizzle's `drizzle.__drizzle_migrations`, so databases previously migrated by drizzle-kit are recognised and never re-applied. Use `pnpm db:generate` (drizzle-kit) to author migrations; a CI `fresh-migration` job guards a from-zero apply. (Fixed in #22.)

**Local DB:** `docker compose up -d db` (Postgres; host port via `DB_PORT`, default 5432). Copy `.env.example` → `.env` first. Full stack (app + db, runs migrations on start): `docker compose up --build`.

## What Toastily is

An open-source, **club-agnostic** website platform for Toastmasters clubs. It ships generic and is configured per-club via the database — the first deployment is the CN Collaborators club, but the repo is **public on GitHub**, so this is a hard architectural rule, not a preference:

- **No club-specific data in code.** Club name, schedule, branding, roles, agenda templates, email/QR settings all live in the `settings` table or seed data, never hard-coded.
- **No secrets in code.** Resend keys, DB URL, session secret, etc. come from environment variables (injected by Coolify) or the admin-only `settings` table.

## Planned stack

## Architecture rules that span multiple features

These cut across the codebase; getting them wrong in one place breaks consistency everywhere.

- **Bilingual (EN/FR) is end-to-end, two distinct mechanisms — don't conflate them:**
  - *UI strings* → `@nuxtjs/i18n` locale files.
  - *User-generated content* (News, editable landing content blocks) → **paired `*_en` / `*_fr` columns**, both required before publishing.
  - *DB-managed labels* (meeting roles, executive positions, agenda items, holidays) also carry `*_en` / `*_fr` fields.
- **RBAC is layered.** Base account status is a linear ladder `guest → member → officer → admin` (officer implies member; only members can be officers). On top of that, two capabilities are **delegable grants** (content-edit, calendar-manage) stored separately, and some authority is **contextual to a meeting** (e.g. the meeting's Sergeant-at-Arms or Toastmaster can open/close voting and view results — independent of executive rank). Enforce all of this in server middleware, not just the UI. See the permissions matrix in the PRD §3.
- **Roles are data, not enums.** Meeting roles and executive positions are DB rows that admins edit; never hard-code their names or hard-code business logic against a specific role label.
- **Participants may be a member OR a guest.** Role signups, speeches (presenter/evaluator), and check-ins all support either a `user_id` or a free-text `guest_name`. Members self-sign-up when logged in; officers may assign a guest (which reveals a manual Name field). Model both paths everywhere participation is recorded.
- **History is first-class.** Role/executive assignments, signups, speeches, and votes are kept over time to power the member-visible participation tracking (PRD §11) and role-allocation history — prefer append/temporal records over destructive updates for these.
- **Agenda generation is template-driven.** An agenda = an ordered list of timed agenda items (DB-managed template, each item optionally bound to a meeting role); generating fills each item from current signups. The on-screen `/agenda` and the downloadable PDF render the same template.
